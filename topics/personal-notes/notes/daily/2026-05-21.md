### Three local-DB drift gotchas surfaced and captured

Bring-up surfaced multiple production-vs-local mismatches that future bring-ups will hit. All now documented in the runbook:

1. **`inframap_customer` missing `endpoint_stage` / `queue_stage` columns.** Production has them from a manual ALTER; migration `0539_historicalcustomer_endpoint_stage_and_more` only patches `inframap_historicalcustomer`. Fix: `ALTER TABLE inframap_customer ADD COLUMN IF NOT EXISTS endpoint_stage varchar(20) NOT NULL DEFAULT 'prod'` (and `queue_stage`). This is a bring-up step, not a one-off — every fresh restore needs it until a backfill migration lands.
2. **`ConnectorVersion.release_date` is NOT NULL with `auto_created=True` metadata flag.** The flag does NOT auto-populate values; `get_or_create(tag=...)` without `defaults={"release_date": ...}` IntegrityErrors. Fixed in `deploy_branch` endpoint by passing `defaults={"release_date": timezone.now().date(), "display_name": f"Custom branch: {image_tag}"}`.
3. **`ConnectorController` constructor takes 6+ positional args.** Use the `from_customer(customer, ecs_client)` classmethod instead — that's the canonical factory and engages the `settings.STAGE != "prod"` safety short-circuit cleanly. Switched in both `deploy_branch` and `_revert_customer_to_pre_custom_phase` helper.

### Critical post-deploy: missing auth on `/api/custom_branches/` GET routes

Live smoke after dev deploy returned HTTP 200 on `GET /api/custom_branches/` instead of the expected 403. Root cause: `CustomBranchViewSet` was extending `viewsets.ModelViewSet` directly instead of the project's `StandardViewSet`, which sets `permission_classes = [IsAuthenticated]` + the `SocialTokenAuthentication / SessionAuthentication / TokenAuthenticationStrict` stack. POST routes were already blocked at the session/CSRF layer and the dev DB had zero `CustomBranch` rows, so no data actually leaked — but this was a clean security hole that would have bitten when CI bots started registering real branches. One-line fix: swap base class. Shipped as [PR #2449](https://github.com/aegissystems/actuate_admin/pull/2449), confirmed live endpoint flipped to 403 within ~30s of ArgoCD sync. **Takeaway:** always extend the project's `StandardViewSet` (or document why not), not raw DRF viewsets. Worth lifting into a CLAUDE.md note for the repo if it isn't already.

### Cleanup orchestrator — the operational point

The whole §29 expansion was driven by realizing the original 3-endpoint per-customer surface didn't solve the lingering-site problem. Today: when a vms-connector feature branch merges, sites pinned to its image tag silently crash-loop on missing-image until someone notices. The new `delete_branch` cleanup orchestrator (`POST /api/custom_branches/{tag}/delete/`) walks all customers with `deployment_phase=CUSTOM` AND `connector_version.tag = <branch>`, calls the shared `_revert_customer_to_pre_custom_phase` helper on each, writes audit events, and marks the branch deleted. Idempotent (returns noop summary if already deleted), partial-failure tolerant (records per-customer errors in summary but continues). This is the leverage that justifies the whole expanded scope.

### Methodology upgrade: ERROR-vs-WARN evaluation in fleet triage (separate session — fleet-RED track)

- **New rule** (saved to memory as [[feedback-error-vs-warn-evaluation]]): every recurring ERROR log pattern in fleet triage should be evaluated against the test "can we (the team) act on this?" If the answer is no — known customer-side flake, expected-rare transient that's already handled with a successful retry, etc. — the emission site should be downgraded to WARN. "Close as observation" is no longer a valid leaf state; it now produces a 1-line follow-up TODO.
- **Rationale**: persistent customer-side ERRORs drown real signal in `fleet_error_top15` and inflate `repo_ci_failure_rate_pct` and related dashboards. The §18 VPA "already exists, patching" sub-item already operationalizes this for one path — generalize the pattern.
- **First application (this session)**: the DW auth `except Exception` arms at the 4 sites. PR #1671 narrowed the cohort by guarding `JSONDecodeError` specifically, but the surrounding catch-all in the same retry loop is still ERROR. The fix is literally `logging.error(...)` → `logging.warning(...)` × 4. Filed as a §20 sub-item.

**Bug shape:** the schedule's `s3://actuate-settings/<deployment_id>/settings.json` had `streams.<name>.threat.metrics = {}` and `raw_metrics = {}`. `billing_emit` called `HttpOnlyAdminDAO.get_product_by_metrics({})` → DAO early-returned `[]` (correct optimization, skip useless admin call) → billing-emit silently fired `misconfigured` fallback per stream. Had been bleeding signal for weeks because step 3 emitted no log.

**Root cause (admin-side):** `actuate_admin` commit `627af2c2` (Dec 2025) changed `list_metrics_by_schedule` to intersect `stream.stream_metrics` with `schedule.schedule_product_metrics(enabled=True).metric_label_ids`. When the intersection is empty, the generator silently wrote `metrics: {}` to S3. Fired on schedules where the schedule's enabled product metrics don't share metric_label_ids with the stream's stream_metrics.

**S3 audit (using `ait` for the first time as an investigation tool):** 217 AP schedules in `actuate-settings`, **139 broken** (64%) — but clustered: 75% last generated in 2025-07, only 3 are active cronjobs (37837 + 41070 still firing; 46767 self-recovered when admin regenerated its settings at 14:48 UTC during the investigation). The 136 dormant broken are mostly `model_name=Dummy` test schedules; not customer-impacting.

**Two-layer guard shipped:**

1. **Connector-side detection guard** — [vms-connector PR #1712](https://github.com/aegissystems/vms-connector/pull/1712), MERGED to `stage`. WARN log with `empty_metrics_dict` token in `HttpOnlyAdminDAO.get_product_by_metrics` when input metrics dict is empty. 3 direct unit tests pinning the log token. Behavior unchanged (still `return []`); only adds visibility.
2. **Admin-side upstream fix** — `actuate_admin` branch `fix/ap-empty-metrics-fallback`, **committed locally, not yet pushed** (paused at user's request for `/session-wrap` before continuing). `list_metrics_by_schedule` now logs WARN (`ap_metrics_fallback_all`) and falls back to `stream.stream_metrics.all().order_by("label__order")` when the schedule_product_metrics intersection is empty. 2 unit tests added covering both branches.

**Dashboard signals added (in `~/.claude/skills/dashboard-check/config/signals.json`):**

- `ap_empty_metrics_warn_6h` — matches `empty_metrics_dict` OR `no products configured` in Connector-EKS pods. Yellow >0, red >50. Currently ~180/6h (red); will drop to 0 once admin fix lands.
- `ap_metrics_admin_fallback_24h` — matches `ap_metrics_fallback_all` in `prod_camera_admin` container. The admin-side warning surfaces data inconsistencies that need per-schedule cleanup (schedule_product_metrics missing or stream_metrics mismatch).

**Investigation methodology takeaway:** the `ait` tool (`actuate-integration-tools` CLI) handled this end-to-end — `ait detections <deployment_id>` for schedule settings inspection, `ait history` for S3 version timeline, custom one-shot script using `ait`'s `fetch_settings`/`list_settings_versions` for the 217-schedule audit. First investigation that didn't need any direct S3 boto3 from scratch. Generalizes well — any "settings-divergence" investigation can probably lean on `ait` for the inspection phase.
