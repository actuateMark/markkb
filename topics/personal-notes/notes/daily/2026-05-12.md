### Firebat `/app/kb/query` rebuild — job pattern + SSE + auto-archive

Triggered by a real 504 on the form: synchronous FastAPI handler used `asyncio.wait_for(proc.communicate(), timeout=150)`, killed claude, and discarded the in-memory stdout buffer — the ~400 KB session was already billed by Anthropic and recoverable only via the JSONL transcript on disk. Committed as `9cd29df` (laptop, `local_network_scripts/main`).

- **Lifecycle decoupled from HTTP request.** POST spawns `~/.local/bin/kb-job-runner` via POSIX double-fork (setsid in first child, grandchild reparented to `systemd --user` pid 1224 — no zombies in uvicorn), then 303s to `/app/kb/query/{id}`. Browser closes / 504s no longer kill the underlying claude run; tokens already spent are no longer thrown away.
- **Stream-json → SSE pipe.** Runner writes `--output-format stream-json --verbose --include-partial-messages` to `~/.local/state/kb-jobs/{id}/stdout`. SSE handler tails the file, parses events into `status` / `text-delta` (token-by-token) / `text` / `result` / `archived` / `done` named SSE events. Heartbeat every 15 s, replay from offset 0 on every connection (refresh-safe). Caddy passes SSE through without buffering as long as `Content-Type: text/event-stream` is set; the explicit `X-Accel-Buffering: no` is belt-and-suspenders.
- **Reaper service.** `kb-jobs-reap.timer` every 5 min: SIGTERM at 10 min, SIGKILL at 13 min (writes synthetic `status.json` on the kill path since bash dies before reaching its own write), purges job dirs > 24 h.
- **Auto-archive to KB.** On `exit 0`, runner invokes `~/.local/bin/kb-job-archive` which walks the stream-json's tool_use file paths, tallies `topics/<X>/...` hits (count, tiebreak by cumulative path length), slugifies the question, and writes a synthesis at `topics/<topic>/notes/syntheses/<date>_kb-ask_<slug>.md` with `kb-bot` frontmatter and a Provenance footer (model, duration, cost, job/session IDs). Idempotent via `meta.json.archive_path`. Verified across three live runs: "what is mork-firebat" → compute-fleet, autopatrol question → autopatrol, infra question with no KB hits → personal-notes fallback.
- **Live archive notice via the same pipe.** Archive script emits one synthetic `{"type":"archive",...}` line; runner appends it to `stdout` *before* writing `status.json`, so the SSE handler picks it up in normal tailing and the browser renders "saved to KB: …" before the `done` event fires. `meta.json` carries archive metadata for the server-rendered refresh path.
- **VAULT_PATH ambiguity bit once.** Actuate convention has `VAULT_PATH=~/Documents/worklog` (parent of the KB), but my script first assumed it pointed at the KB root and wrote a file to `~/Documents/worklog/topics/...`. Auto-detection now checks for `topics/` first, descends into `knowledgebase/` if missing. Worth applying the same pattern to any future tool that reads `VAULT_PATH` — the convention is genuinely ambiguous.
- **Spiritual relative of §29.** This is "do it from a Claude session" via a web form rather than an admin endpoint — same shape: detached job, stream progress, persist artifact. Worth holding in mind when §29's endpoint family solidifies; the runner/reaper/SSE plumbing is portable.

## Soak-script post-mortem

**The "argparse exit 2" diagnosis from this morning was wrong.** Both `pr-1688-soak-t1h.service` (fired 2026-05-11T21:00Z) and `pr-1688-soak-tonight.service` (fired 2026-05-12T02:00Z) ran to completion successfully — each produced a full check report with all 8 signals evaluated, wrote markdown to that date's daily note, and persisted a JSON state file under `~/.local/state/minipc-tasks/pr-1688-soak/`. They exited with status 2 because **the script's verdict-as-exit-code design** mapped RED → `sys.exit(2)`:

```python
sys.exit({"GREEN": 0, "YELLOW": 1, "RED": 2}[overall])
```

systemd's LSB exit-code reporting then surfaced this as `status=2/INVALIDARGUMENT` in `systemctl status` output, even though `argparse` never ran. The morning post-mortem latched onto the "INVALIDARGUMENT" label and assumed a missing flag definition — neither of which was true. RED verdicts were genuine (pre-restart fleet ERROR counts were elevated due to the 4 stale DW pods).

**Fix applied at `local_network_scripts/files/pr-1688-soak-check.sh` (lines 279–286):** the script now always `sys.exit(0)` on successful run. Verdict is surfaced via:
- JSON state file (`~/.local/state/minipc-tasks/pr-1688-soak/<label>-<ts>.json`)
- Appended markdown in the daily note (already working)

Future monitoring that wants verdict-driven alerting should read the JSON or grep the daily-note section, **not** check `systemctl is-failed`. The earlier design conflated "script crashed" with "verdict bad" — exactly what bit us this morning.

Validated at 14:39Z by manually firing `pr-1688-soak-t1h.service` on Firebat with the patched script:
- `Main PID: 2109309 (code=exited, status=0/SUCCESS)` ✓
- `systemctl is-failed` returns `inactive` ✓
- New autonomous report appended to today's daily note (PR #1688-specific signals all GREEN; `connector_pod_errors_1h` returned RED at 12,722/hr but this is fleet-baseline drift unrelated to #1688 — threshold of 500 was calibrated against a much-lower historical baseline and is stale).

**Generalizable lesson worth carrying forward:** verdict-as-exit-code on a oneshot systemd unit is a footgun. The mis-diagnosis cost ~half a session today. Default pattern for any future Tier-1 verification script: process exit = `did the script complete?`, verdict = artifacts on disk. If you want systemd-level alerting on verdict, layer a separate "verdict-monitor" timer that reads the JSON.
