## Notes / Learnings

- **`tailscale debug prefs`, not `tailscale status --json`**, is where `RunSSH` lives. `--json` exposes peer state and capabilities; the *local node's* `up` flags are in `debug prefs`. Lesson: when probing for "is flag X on?", `debug prefs` is the source of truth.
- **`tailscale up` requires `--reset` when transitioning off a non-default flag.** Previously-passed flags are sticky; not re-passing them errors with "changing settings via 'tailscale up' requires mentioning all non-default flags." `--reset` returns all non-passed flags to defaults, which is what we want when migrating off `--ssh`.
- **Disabling Tailscale SSH severs the in-flight session** if you're connected via Tailscale SSH. The phase-15 block now swallows the broken-pipe exit and re-verifies state in a fresh connection.
- **Heredocs are not paste-safe for secret tokens.** Some terminals add leading whitespace on paste, and `EOF` only terminates at column 0. Updated the shell-snippet-format memory rule (rule 2) to mandate the editor approach (`umask 077 && touch <f> && chmod 600 <f> && ${EDITOR:-nano} <f>`) for any single-line content the user pastes.
- **Don't fingerprint a credential file by sampling its contents.** `head -c 50 file` to "verify it's the right file" prints token bytes to terminal scrollback. Use `wc -c file && stat -c %y file` (size+mtime) or `sha256sum file | cut -c1-12` (content-tied hash) instead.
- **Broad gitignore globs cause silent file exclusion.** Always re-run `git status` after editing `.gitignore` to confirm intended files still appear, and `git check-ignore -v $(git ls-files)` for tracked-file false positives.
- **Three "patrol-style" integrations share `-chm-cronjob` container suffix** — only AutoPatrol runs ML; VCH and CHM are health-check only, with intentionally empty `models` arrays. Any NRQL/dashboard filter that targets ML-config issues must filter to `container_name LIKE '%-autopatrol-%'`. Bare `%-chm-cronjob%` produces 10-15× false-positive sites. Tracked: AUTO-566.
- **VCH `/Patrols/` is event-driven, AutoPatrol's is continuously polled.** This is a category mismatch the cleanup pipeline didn't account for — VCH legitimately empty between events, but the connector emitted `no_patrols` cleanup signals anyway. Caused 102/102 VCH cleanup signals over 7d to come from a single test tenant. Real disable signal still flows via puller-level `make_puller_error_callback` → `site_disabled`/`deleted`. Lesson: validate the upstream API's emission semantics per integration before reusing a "no events = stale" predicate.
- **Cleanup Lambda's tenant cascade uses `>=` not `==` threshold trigger.** Admin endpoint is idempotent so retry cost is ~zero, and the wider trigger window means transient admin/Immix outages self-heal at the next cycle without needing manual intervention.
- **`schedule.delete()` not `.save()` for soft-delete in admin.** `delete()` calls `soft_delete + undeploy()` per `autopatrol_schedule_model.py:391-394`. `.save()` alone leaves the K8s cronjob running, which means cleanup re-emits and you're in an infinite loop. Same shape applies whenever soft-delete is paired with a deploy-side cleanup hook.
- **Per-bucket lifecycle audits and structural rework belong in different work tracks.** S3 cost-reduction synthesis crystallized: 11 standalone levers (~$3-20k/yr) vs 1 fleet-arch-coupled lever (~$30-60k/yr). Conflating them creates pressure to make standalone decisions on coupled items. Now tracked separately under ENG-183 parent + sub-tasks.
