### §21 — VCH SIGTERM billing-event fix verified on prod + hardened with tooling/process

- **Verification (prod GREEN, stage YELLOW for unrelated reason).** PR #1667/#1668 verified on prod via `nrql-investigator`: 38/38 SIGTERM'd VCH pods in the prior 24h emitted `site_product_ended` events (vs 0/N pre-fix). End-to-end timeline on pod `r92hj` showed billing fire 385ms after SIGTERM, well inside the 10s grace period. Total VCH: 3476 events / 328 pods / 24h = ~10.6 events/run. Zero new ERROR fingerprints vs pre-deploy baseline. Stage was YELLOW because all stage VCH pods crash on startup with a pre-existing `KeyError: 'monitoring'` — surfaced as new §22.
- **Dashboard signal — 5-signal suite landed on Firebat.** Per-integration `site_product_ended` emission tracked: `vch_billing_emit_6h` (~870 baseline), `chm_billing_emit_6h` (~31K), `autopatrol_billing_emit_6h` (~360), `analytics_billing_emit_6h` (~149K), `fleet_billing_emit_6h` (~181K). All green on first run with values matching the agent's baseline measurements. Per-class split lets us attribute drops to a specific integration; fleet aggregate catches event-handler/SQS/ELB outages. Primary regression detector is `silent_drop_inverse` against rolling 7d baseline (activates ~2026-05-10 once history accumulates). Edited at `~/.claude/skills/dashboard-check/config/signals.json` (88 → 93 signals).
- **Acceptance criterion codified.** New "Billing & Lifecycle Invariants (vms-connector-specific)" section added to [[code-review-checklist]] and mirrored as "Billing-event lifecycle invariant" review heuristic in `vms-connector/CLAUDE.md`. Captures the invariant ("every run emits `site_product_ended` exactly once per camera per run, on every exit path"), the per-stream tracking pattern from `VCHCamera._send_product_ended_events_once`, the unit-test requirement (double-call + interrupt path), and references PR #1663 (patrol managers) + PR #1667 (VCH SIGTERM) as the case studies.
- **Architecture finding — billing-emit logging path.** `event_listener.py:44` (`actuate_event_listener` library) emits `Sending event_info: ...` log line on every successful SQS send to `event_queue_analytics.fifo`. The `event_type: 'site_product_ended'` field is in plaintext (the pickled blob is base64-encoded but `event_type` is unencoded by `encode_v3_data`). NRQL filter `message LIKE '%site_product_ended%'` cleanly proxies emission count.
- **Threshold-calibration learning.** Started with `Sending event_info` AND `site_product_ended` filter pair, got 1016 events vs the agent's 3476 — 3.4x discrepancy. Dropped the additional `Sending event_info` filter and matched the agent's count. Probable cause: NR log line truncation on the verbose `Sending event_info: {...}` JSON dump means some matches lose the prefix. Worth remembering for future log-based signals: prefer the most distinctive substring as the sole filter; layered filters can drop legitimate matches if individual log lines are long enough to trip truncation.
