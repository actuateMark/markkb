## Notes / Learnings

- **The `rtsp_camera_simulator/` dir is gitignored.** Discovered this when trying to add v4l2 support to `camera_simulator.py` — the modifications would have silently no-committed. The dir lives locally only; new test fixtures need to live elsewhere in the tree. Pivoted to a self-contained `test_settings/webcam_rtsp/` fixture using `bluenviron/mediamtx:latest-ffmpeg` instead — cleaner anyway (one container with bundled ffmpeg + RTSP server, no host ffmpeg dependency, no modifications to existing gitignored code).

- **`test_settings/*` is also default-deny with per-file `!` allowlist negations.** New test-settings files need an entry in `.gitignore`. Easy miss because `git status` shows them as untracked rather than ignored unless you specifically ask. Same trap likely caught new test settings on other branches.

- **memray flamegraph render can hang indefinitely on larger `output.bin` files.** Today's 7.9 MB bin sat at process state `S` with 0% CPU for 17 min until I killed it manually. The session orchestrator's `subprocess.run(...)` for the flamegraph render has no timeout, so a hung renderer hangs the whole session. Two fixes for tomorrow: (a) timeout the render call and fall back to skipping the HTML; (b) the same root cause apparently breaks `memray stats --json`, because today's `report.md` came up with no allocation-hotspots tables. Filed in the handoff.

- **`uv run --with memray` resolves deps from the cwd's project.** When the wrapper runs the session from `actuate-instrumentation/` cwd but the memray runner shells out from `vms-connector/` cwd, uv has to re-resolve memray in the connector's project context. That needs the CodeArtifact token in scope, hence the wrapper exports `UV_INDEX_CODEARTIFACT_USERNAME` / `PASSWORD` (plus `PIP_INDEX_URL` / `PIP_EXTRA_INDEX_URL` because pre-commit subprocesses use pip not uv). All of this is documented in the wrapper.

- **Worktrees survive between sessions only if `/tmp` survives.** The `/tmp/vms-connector-cv2` worktree from 2026-05-15 was gone today after `/tmp` was cleared; `git worktree prune` + recreate took 5 seconds. Worth knowing: long-lived worktrees should live under `~/work/` or `~/worktrees/`, not `/tmp/`.

- **Today's session was sibling-stream to PR #60 cutover.** PR #60's Monday cutover was the primary workstream of the day; §30 ran in a parallel session and didn't compete for the same files. Handoff captured separately so neither bleeds into the other's daily-wrap.

- **Customer-side validation gates are calendar blockers, not technical ones.** PR #60 cleared every pre-merge gate on our side; the cutover paused on partner sign-off. Once the verification surface is current and the three out-of-band gates (Lambda authorizer DDB partner key, `WEBHOOK_SECRET` on Confluence-notifier Lambda, partner cutover timing) are confirmed, "merge when ready" is the right state to leave it in.

- **PR #82 was a near-miss prod outage.** Someone on the team noticed today that `terraform/environments/388576304176/eu-west-1/{dev,prod}.tfvars` were missing the v5 model-version + endpoint env vars. The dev-EU API was already 502'ing, and prod-EU would have followed the moment PR #60 landed. 22-line fix, merged 14:41Z, picked up by PR #60's CI re-run. Lesson: env-var coverage across all `(region, account)` tfvars is part of "release-ready" — the dev us-west-2 verification we ran isn't a stand-in for region/env coverage. Worth a checklist line in the `/validate-release` flow for inference-api.

- **Stale verification scripts are silent regressions.** The `/tmp/v5_motion_pertag_test.py` crashed on `KeyError: 'confidence'` and the `/tmp/v5_detect_motion_verify.py` reported 3/9 FAILs — both were stale-script symptoms (opt-in confidence + PR #77 `reason`/`error` split), NOT API regressions. Could easily have been read the other way around in a hurry. Catch: when a verification script "fails," diff the assertion against the current schema before chasing the API. The fix landed in commit `92b574e`, so future verifications start from a current baseline.

- **§24 LLM shop — five pre-batch quality gates shipped to `kb_deep_intake`** (commit `2efb7df` in `local_network_scripts`). Diagnosed the 2026-05-11 batch where 12 of 17 drafts were thin/off-topic, classified failure modes into six patterns (thin marketing pages, off-topic for Actuate, GitHub README boilerplate, duplicate URLs, miscategorized topics, code-inferred experiments), and added: (1) **min-content gate in parser** — bails `too_thin` if total <1500 chars or <2 substantive sections, zero LLM cost; (2) **topic-description map in planner prompt** — `kb-batch-submit` walks every topic's `_summary.md` and feeds slug→description to the model so "integrations" reads as "Actuate vendor packages only" instead of a keyword match; (3) **relevance pre-check** (new `relevance.py`) — single SYCL-14B rubric call returning `high|medium|low`, rejects low, flags medium, fail-open on backend error; (4) **post-plan substance check** — bails if plan has 0 sections, 1 section <200 target words, or >70% of source sections skipped; (5) **submit-side URL dedup** — refuses URLs already present in any topic's `_research-inbox/`. Deployed via scp to `actuate@npu-server:~/llm-shop/`. Live thin-page test against `http://www.live555.com/` short-circuited at `parser_too_thin` in 3.3s with zero LLM calls — gate behaving exactly as designed.
