### actuate-libraries release rule — corrected understanding

Initial reading of "we don't merge dev versions" was that the version on the branch needed to be clean before merge. **Corrected:** the actual lever is the squash commit message. `Publish Dev` re-stamps `pyproject.toml` to `X.Y.Z.devN+slug` on every push to a feature branch (no commit-message escape works). `bump-stable.yaml` runs on merge to main, scans the merge commit, and **overwrites** the dev version with the next clean stable — but only if the squash commit:

1. Contains `[patch:<package>]` (or `[minor:<package>]` / `[major:<package>]`) — triggers bump-stable
2. Does NOT contain `[no ci]` / `[skip ci]` / etc. — those are GitHub-builtin tokens that suppress all workflows on the merge commit, including `Publish Stable`. Default `gh pr merge --squash` drags every commit body into the squash, including the auto-`[no ci] Bump versions for: <pkg>` commits added by `Publish Dev`. **Always pass explicit `--subject` and `--body`** (or edit in the GitHub UI) — the default behavior is unsafe.

This bit the project on PR #341 on 2026-04-22 (squash dragged `[no ci]` into body, suppressed Publish Stable, left main with the dev string). Was on track to bite again here on PR #342 until I explicitly built the squash subject/body. Verified via post-merge: bump-stable ran, version on main is clean `1.17.16`, stable artifact published, connector pin bump installed it cleanly.

Encoded the corrected rule in three places:
- Global `~/.claude/CLAUDE.md` Git Safety section
- `feedback_library_no_dev_versions.md` auto-memory + index entry in `MEMORY.md`
- KB topic `topics/actuate-libraries/_summary.md` "Pre-merge release rules" section

(First-pass version of the rule was wrong — said "bump version on branch to clean before merge". Corrected on second pass once I dug into `bump-dev.yaml` + `bump-stable.yaml` and saw the workflow re-stamps every push regardless of commit message. The branch state is harmless; only the squash matters.)

### 2026-05-02 InferenceAPI outage — root cause + Tier-2/3 detection signals

**Root cause confirmed (via prod AWS investigation).** ECR lifecycle policy on `inference-api-lambda-prod` had a rule:
```json
{"selection": {"tagStatus": "tagged", "tagPatternList": [".*"], "countType": "imageCountMoreThan", "countNumber": 50}, "action": {"type": "expire"}}
```
`tagPatternList: [".*"]` matches every tagged image; the count rule then prunes the oldest past 50 — without consulting whether any are referenced by a deployed Lambda's `Code.ImageUri` (digest-pinned). Lambda cold-starts then failed with InternalServerError; existing warm containers kept working until they recycled, so the outage degraded gradually rather than all at once. The post-incident fix on us-west-2 was a full `delete-lifecycle-policy` (no replacement). No Lambda log group was created during the incident (`/aws/lambda/InferenceAPI-prod` was missing) because Lambda never reached the runtime — itself a meta-detection signal.

**Audit caught 4 of 6 dangerous policies still live.** Even though the user was told "policies have been removed", a fresh re-check showed the fix was applied to **only one repo** — `us-west-2 / inference-api-lambda-prod`. Five sibling repos still had the same broken rule, including `eu-west-1 / inference-api-lambda-prod` (the same setup as the May 2 victim, just in EU). After confirming with the user, applied `delete-lifecycle-policy` to the 4 remaining (with backups at `/tmp/ecr-policy-backups/*.json`). Final account-wide sweep across all 297 repos in 3 regions: zero dangerous policies remain.

**Tier 2 — preventive audit signal `~/bin/ecr-lifecycle-audit`.** New Firebat-resident Tier-1 script (per [[2026-04-30_three-tier-routine-check-pattern]]). Walks 3 workload surfaces:
- **Lambda** via boto3 (10 functions across regions)
- **ECS** via boto3 (35 containers across ~80 customer connector clusters)
- **EKS** via local Argo manifests at `~/work/kubernetes-deployments/` (122 deployments — handles literal ECR URIs and Helm-templated `{{ .Values.X.ecrRepository }}/<repo>:<tag>` refs, emitting one workload per region for templated refs since both inference EKS clusters apply the same chart bundle with different region-scoped `ecrRepository` values)

For each `(region, repo)` group: fetches lifecycle policy + image set once, runs the simulator (any image past `countNumber` rank in a `.*`-pattern rule is doomed), resolves each workload's image (digest if pinned, else tag → digest via the repo's image set), flags any where deployed digest is in the doomed set. Cross-account images and templated-repos that don't exist in a given region are silently skipped. Two sink observations per run + a markdown digest. Daily systemd timer at 13:00 UTC. Two new IAM inline-policy statements added to `dashboard-check-rolesanywhere`: ECS list/describe + ECR get-lifecycle-policy/describe-images. Canonical at `local_network_scripts/files/iam-policy-ecr-lifecycle-audit.json` v2.

- `ecr_pruning_risk_count` (component `ecr-lifecycle`) — RED on any non-zero count.
- `ecr_dangerous_policies` — FACET dict of `region/repo` → image_count for any repo with `.*`-pattern expire rule. Yellow on any presence, red if also at risk.

**Tier 3 — reactive `cw_metric` collectors.** Added `cw_metric_inference_api_prod_errors_uw2` + `_euw1` to `collect.py` (Sum(AWS/Lambda Errors) over 1h per region). Two new signals `inferenceapi_prod_errors_us_west_2` + `_eu_west_1` (component `inference-api`), threshold red >50 errors/h. Both regions report 0.0 currently. Catches anything that spikes Lambda errors — image-pull failures, unhandled exceptions, OOMs, init-time crashes — within ≤1h instead of the May 2 hours-long-detection window.

**"Would-have-caught" verification.** Unit-tested simulator with synthetic 60-image-repo + the exact `tagPatternList: [".*"]` + `imageCountMoreThan: 50` rule. Doomed set = oldest 10.
