### Dashboard-check null-signal repair — four failure modes in one sweep

**Mode 1 — NRQL transient (no action).** `connector_dummy_model_fallback_24h` (SERVER_ERROR `NRDB:1107005`) and `nonetype_unpack_top_facet` (TIMEOUT `NRDB:1109`) hit NR-side hiccups intermittently. Existing `try/except → None` handling is correct; the visibility problem (next mode) is what made these look catastrophic instead of transient.

**Mode 2 — Renderer fallback bug (the sneaky one).** `render.py::_latest_sink_observation` explicitly skipped ALL `source_skill == "dashboard-check"` rows when computing the "last good value" fallback. Intent appeared to be "don't double-count the current run" but it nuked ALL prior history from the same source skill. So when today's NRQL flaked, the renderer found no prior-run sink history → wrote `value=None, status=error, "no sink history"` → that became the most recent sink entry → next run's fallback also returned nothing. A flake became a sticky failure. Fix: include dashboard-check rows in the fallback search, but skip rows whose own state was None/error so we genuinely fall back to last-good-value.

**Mode 3 — Missing dispatchers (config bug).** Three `billing_*` signals declared `source: minipc_local` with shell `command:` strings in signals.json, but `collect.py::MINIPC_LOCAL_DISPATCH` is a function-registry — not a shell-command runner. So they printed `"skip ... no minipc_local dispatcher"` and returned None. Fix: added 3 Python functions (`local_billing_production_unbilled_cams`, `_reconcile_residual`, `_reconcile_freshness`) parsing the daily JSON at `~/.local/state/minipc-tasks/billing/reconciliation-YYYY-MM.json`. All return real values now (2331 / 0 / 1).

**Mode 4 — Cross-region IAM AccessDenied (informational demote).** `onboarder_*_eu_west_1` from `autopatrol-cleanup-check` failing with `FilterLogEvents AccessDeniedException` — the `dashboard-check` Roles Anywhere role doesn't have eu-west-1 CloudWatch Logs permission. Likely because the EU onboarder Lambda lives in account 558106312574, not the primary. Real fix is IAM/Terraform; pragmatic fix is to demote to `status=informational` with the AWS error in notes so the signal stops poisoning the RED bucket.

**Process learning — laptop↔Firebat config drift is non-trivial.** Friday's AP summary-rate NRQL fix had been sitting in the laptop's `signals.json` but never synced. Today's `sync-claude-skills.sh --check` would have caught it. Adding a "check sync diff at end of session" loop to the EOD ritual would have prevented the 4-day false-RED.

### libav warmup-frame artifact — full root-cause synthesis written

After 4 days of incremental fixes the Eyeforce VCH validation cycle landed end-to-end. The root cause wasn't camera-side blank video — it was a **libav decoder artifact** where the first decoded frame after a fresh `av.open()` on a fragmented MP4 fragment is an uninitialized zero-pixel buffer when the fragment starts mid-GOP (P-frame without its reference I-frame in the new decoder context). Confirmed empirically by MD5-identical pure-black 720x480 JPEGs across multiple cameras' "current_image" saves. Three-part fix landed across vms-connector + actuate-libraries: multi-frame scoring + drain-after-`thread.join()` + per-fragment warmup-skip in healthcheck mode. Full story: [[2026-05-18_libav-decoder-warmup-frame-fix]].
