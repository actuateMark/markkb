### §13. Subagent + cron MCP-bypass auth flow

**Priority:** high — surfaced 2026-04-27 morning routine (3 NR-touching subagents stalled; both cron jobs failed)
**Tickets:** *(personal infra — no ticket)*

**Trigger:** 2026-04-27 — daily-scope fan-out launched 4 subagents; 3 of them stalled indefinitely after spawning interactive NR auth prompts the subagent couldn't satisfy. Same morning, `overnight-check.service` and `jira-sync.service` both wrote FAILED stubs because their headless `claude -p` invocations couldn't reach NR or Atlassian MCP either. `/repo-scan` worked because it only used `gh` CLI.

#### Problem

MCP servers (NR, Atlassian) authenticate per-Claude-Code-subprocess. Each subagent and each headless cron run gets a fresh MCP session that must complete an interactive auth handshake. Subagents can't satisfy interactive prompts — they hang. Cron `claude -p` runs without MCP creds — they fail. This blocks:
- daily-scope morning fan-out (NR overnight, cleanup-lambda check, dashboard-check)
- Scheduled overnight-check synthesis (cron job)
- Scheduled jira-sync (cron job, Atlassian)
- Future minipc-side scheduled runs (§11e)

#### Approach (Option A from 2026-04-27 daily-scope review)

Bypass MCP for headless / subagent paths. Use REST API wrappers backed by API keys on disk. Subagents and cron call them via Bash; parent-context Claude continues to use MCP interactively. For 2026-04-27 only the parent ran NR queries directly as a one-off bypass (Option C); structural fix is this workstream.

## Notes / Learnings

- **MCP auth is per-subprocess and interactive-only.** Subagents and headless `claude -p` invocations both inherit a fresh MCP session that requires a browser callback to complete OAuth. There's no token-on-disk path. The structural workaround is REST wrappers backed by API keys (`~/.config/<service>/api-key`, 0600). Documented in [[2026-04-27_headless-mcp-bypass]].
- **Connector-45999 emerged as a NEW chronic OOMKill offender** (109/24h, not in yesterday's top-5: 14170, 23422, 23730, 20628, 45010). Seeded as a tomorrow `verify` follow-up; if sustained, escalate as a §3 / §9 follow-up (memory-limit drift candidate per [[2026-04-23_oom-surge-connector-limit-drift]]).
- **`(copy)` suffix is actuate_admin's clone-from-template marker.** Customer 23202's site cluster all carrying that suffix is the strongest signal yet that the 2026-04-17 SSL blast-radius growth (`dev.powerplus.com` issue) was a single customer's bulk-clone deployment, not an unrelated infrastructure shift. File this pattern under fingerprints worth grepping the admin DB for in future SSL incidents.
- **mark-todos was overdue for the rolling-forward refactor.** 1219 lines / 57k tokens — almost half the file was closed `[x]` history accumulating inside still-open workstreams. The new convention (closed sub-items move to that day's daily note immediately on close, enforced by `/daily-wrap` Step 2.7) prevents this from recurring. Snapshot at `_archive-snapshots/` is the safety net.
- **Date attribution worked surprisingly well during the sweep.** Most `[x]` items carried explicit dates in their text (PR merge dates, "DONE 2026-04-22", etc.). Only a handful needed batched-into-2026-04-27 with `(swept)` qualifier — and those were structural items where the close date is genuinely fuzzy.
