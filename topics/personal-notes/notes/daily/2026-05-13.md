## Notes / Learnings

- **Local RTSP simulator (port 8554, docker container `rtsp-camera-simulator`) is the right entry point for §30 profiling work.** The `VMS_CONNECTOR_LOCAL_RTSP.setting.json` had a stale `server_ip: 158.106.215.138` at the top level (unused for `integration_type: rtsp`) — the cameras stream from `127.0.0.1:8554/camera`. Cleaner pattern for future scenarios is to confirm the camera URL before assuming an external VMS server is needed.

- **The 100-second `fps_and_processing_sample_period` in the local RTSP settings is a friction point for short profile runs.** Captured breakdown JSON is empty for any run under ~120 s; the harness's own RSS CSV and the in-probe timer-driven parity dump compensate for this. Filed as a follow-up: harness flag to mutate the settings file's breakdown period for verify/short-session runs.

- **Connector dropped to CPU H.264 decode on this laptop** ("Failed to create cuda decoder context for h264: h264_cuvid"), so the speedscope flamegraph captured during the session run is skewed toward software decode. That's a real signal for the optimization initiative — the production fleet probably hits the same fallback on hosts without `h264_cuvid`, and the cost is visible in the speedscope.

- **Live-bug pattern: shared-snapshot parity testing is non-negotiable when the workload allocates large buffers per frame.** Two back-to-back `tracemalloc.take_snapshot()` calls drifted by 13 MB because a frame-buffer landed between them. The fix generalizes: any "compare two views of process state" experiment must pin the state explicitly, not take two adjacent snapshots and hope they're close. Same pattern will apply to any future memory- or CPU-profile comparisons.

- **`uv run --with memray …` doesn't forward signals.** When the harness sent SIGINT to the uv wrapper, memray under it stayed alive indefinitely (output.bin stopped growing but the process never exited). `start_new_session=True` + `os.killpg(pid, SIGINT)` fixes this cleanly, with a bounded SIGTERM/SIGKILL escalation if memray still won't quit. Pattern to remember for any future `uv run --with`-wrapped long-running subprocess.

- **Local working tree can be hours stale vs `origin/develop` even after `git fetch`.** Burned ~30 min today chasing a "regression" that was actually 5 commits of fixes (PR #63 LegacyError envelope repair, per-frame URL id) that hadn't landed in my checkout. Memory: [[feedback_pull_before_prerelease_verify]]. Always `git pull --ff-only` before any regression run that claims PR-head evidence; quote the SHA in the test summary ("verified against `develop` @ `aa89e68`") so a stale claim is detectable later.

- **PR audit claims need live verification before being transcribed into a PR body.** The auditor was correct *for `origin/develop`* but my local working tree was stale, so two audit findings appeared contradictory. Memory: [[feedback_pr_audit_live_verify]]. Prefer "✅ Live-verified via …" wording in PR bodies over "✅ Per audit" — forces the probe.

- **`v4.py:340` lost `logger.append_keys(request_id=id)` on `infer_intruder_plus_with_vehicle_sequence`.** X-Request-Id header echo is intact (verified live), but New Relic log correlation by `request_id` will stop working on that one endpoint after PR #60 merges to prod. Worth a `chore(v4)` follow-up issue post-merge.

- **The `558106312574 / eu-west-1 / prod` Terraform cell hasn't seen a successful apply in 44 days.** Stuck lock was hiding the fact that every prod-targeting PR run had been failing on that cell. The `deploy-dev` workflow succeeded on the same cell with a different `STAGE` workspace path, making the failure look transient. Lock age + Terraform-version mismatch (1.14.8 in the lock vs 1.11.4 in current CI pin) is a reliable orphan-tell.

- **`tools/v5-test-page/run.sh` now tees server stdout to `/tmp/inference-api-local.log`.** Made the LegacyError envelope diagnosis tractable — direct trace from `v4.py:483` to a Pydantic ValidationError, which is what the auditor had flagged and the stale local code was reproducing. The log file is per-run-truncated; preserve copies before restarting `run.sh` if a particular trace matters.

- **The `sync` workflow's silent-failure mode is broader than just the EDOCS permissions.** The script's main loop has a top-level `except Exception` per file, so any partial failure (rate limit, network blip, malformed page) is logged-and-skipped. The fail-loudly patch (reverted today) is the right shape but should land *with* the credentials fix in #67, not before.

- **Long copy-pasted shell commands with embedded JSON get mangled by terminal wrapping.** Multiple AWS CLI commands today failed when the terminal inserted newlines mid-JSON or split the `--table-name` flag. Switching to bash helper scripts (`/tmp/tf-unlock.sh`) sidesteps the problem cleanly. Pattern for future: for any command longer than ~80 chars with quoted JSON, write a script and tell the user to invoke it.

- **Library-side direct-DB consumers can hide from a service-focused audit.** The original AdminDAO inventory missed `actuate-libraries/actuate-config` (instantiates AdminDAO purely to access its `.admin_api` attribute — a wrapper-only pattern) and `queue_consumer/health_consumer.py` because it grouped findings by consuming service rather than by import location. The scripted re-audit catches them by scanning every `*.py` under `/home/mork/work/`. Pattern for any "where is X used" claim that's about to be published externally: re-run a scan that includes libraries, not just services.

- **Scripted re-audits beat narrative inventories.** Today's discovery pattern: manual inventory has placeholder rows → scripted scan finds 2 unmentioned consumers + 1 wrapper pattern → architectural correction (wireguard claim was misleading; already in admin API) lands. Re-runnable inventory scripts in `/home/mork/work/scripts/` should be the standard hygiene step before publishing any cross-repo claim. The 3 scripts shipped today (admindao-inventory, postgres-direct-callers, admin-api-surface) are a template for similar audits in other workstreams.

- **Mutable Docker image tags break infrastructure-marker rollout verification.** `arm_connector_rearch:latest` mutates in place; can't tell pre/post deploy by image tag alone. Verify by behavior (log rates, error patterns, deploy event TIMESERIES) not by tag string. The rolling-restart agent leaned too hard on tag-set comparison at first; the negative log-rate signature is the correct signal. Applies generally — any time you want to detect "did this restart happen" or "did the new code reach pods," reach for log-based step-change evidence first.

- **Customer model has 6 candidate split seams; Seam A is the highest-leverage near-term cut.** §29's two new fields, AI-184's incoming `customer` FK from Sensitivity, and the recent `f1ad8fcb` regression all cluster in the "deployment configuration" area of Customer.
