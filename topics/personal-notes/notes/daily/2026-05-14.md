## Notes / Learnings

### From the v5 release-prep work

- **S3 bucket names are globally unique.** Creating the same name across 3 (account, region) deploy targets fails on the 2nd; the bucket pattern needs account + region in the name when the same resource is deployed multi-region/multi-account. New pattern: `actuate-inference-api-frame-cache-${stage}-${account}-${region}` (61 chars, under the 63 cap).

- **CI deploy role needs `iam:PutRolePolicy` + `iam:DeleteRolePolicy`** to manage **inline** policies (`aws_iam_role_policy` in Terraform). `iam:AttachRolePolicy` / `DetachRolePolicy` covers **managed** policies only ÔÇö different action set.

- **IAM eventual consistency during Terraform apply.** When the same apply updates the deploy role's policy AND uses the new perms, the existing session token won't see the new perms. Result: 403 on the first attempt, success on rerun. A rerun gets a fresh session token. Two of today's three deploy retries were this exact dance.

- **`s3:ListBucket` is required for canonical 404 on missing keys.** Without it, S3 returns `AccessDenied` (403) for missing objects (it hides key existence from unauthorized listers) instead of `NoSuchKey` (404). Cache implementations that need to distinguish "miss" from "permission failure" must grant `ListBucket` on the bucket (without the `/*` suffix ÔÇö bucket-level action).

- **Cursor's post-merge re-scans surface findings on adjacent unchanged code.** Each round of PRs reset cursor's view of the codebase, and it'd flag pre-existing patterns adjacent to the new changes. Filter cursor comments by `created_at > <merge-timestamp>` strictly ÔÇö anything before is pre-existing context, anything after is a fresh finding from the new scan. (The watcher got this wrong on the first try by treating old comments as "new on HEAD". Fixed by passing the timestamp cutoff explicitly.)

- **`AccessDenied` from S3 for `GetObject` is ambiguous between "no perm to read this object" and "no perm to list this bucket".** First time encountering this it cost an extra debug round. Standard AWS docs reference; worth caching as an entry-point lesson for any future S3-cache feature.

- **GitHub push-event workflows can silently miss firing.** PR #69's merge didn't trigger `unittest.yaml` or `deploy-dev.yaml` on develop (0 workflow runs against `b431af2`). Recovery pattern: push an empty commit (`git commit --allow-empty`) to develop to force a new SHA + fresh workflow trigger. Documented this in the commit message so future bisects don't get confused.

- **The 7081731 vs 3421145 NR account split is still unresolved.** The E2M graphql file targets `7081731` but `list_available_new_relic_accounts` only returns `3421145` (primary) + `4824056` (storage). Either Michael has access to a sub-account no one else does, or the graphql was never applied. Outstanding follow-up.

## Notes / Learnings

- **The point of the suite is the report, not the artifacts.** A 7 MB flamegraph HTML and a 363 KB speedscope JSON aren't *findings*; they're *evidence*. Without the report subcommand, "we have a profiling suite" meant "we have a directory full of blobs". With it, a session is a one-page hotspot read. This is the same lesson the connector's existing tooling missed ÔÇö `cpu_profile.sh` and `memory_profile.sh` produce artifacts you have to manually open.

- **Memray `stats --json` is much more useful than the flamegraph for top-N.** The flamegraph is great for stack-shape exploration but terrible for "give me the top 10 allocators". `memray stats --json -n 10 -o stats.json output.bin` gives `top_allocations_by_size`, `top_allocations_by_count`, `total_bytes_allocated`, and `metadata.peak_memory` ÔÇö exactly what a report needs. Re-running it on an existing `.bin` is cheap, so the report can drop `stats.json` next to the binary and reuse it.

- **`cv2.foo(dst=ÔÇª)` is a generally underused win.** Every cv2 function in the motion-detection path supports a preallocated output buffer; we're paying for fresh numpy allocs per frame across the entire pipeline. The pattern is so repeatable that it's a candidate for a small wrapper helper plus a one-time per-detector buffer-shape initialization ÔÇö not a 50-LOC refactor but a 50-LOC *spread* across one well-bounded module.

- **`expire_items` 11 % CPU is a py-spy `--idle` artifact, not a bug.** The function is `while True: time.sleep(15); if stale: expire()`. It can only show up unless py-spy is told to count sleeping threads. Worth checking `runners/pyspy.py` before chasing it as real work.

- **The connector's `_log_memory_breakdown` interval (100 s) is longer than verify-mode runs.** Both validated sessions captured zero breakdown cycles. Either lengthen runs (cheap but slows verification) or expose the interval via env-var (cleaner; the breakdown can run every 30 s for verify mode). Tracked as a ┬º30 follow-up.
