### Unplanned: PR #77 — `detect_motion` reason/error split + always-present stationary

Triggered by reviewing the verification output: cold-cache calls returned `{"error": "no_previous_frame"}`, but `no_previous_frame` is normal flow on the very first call for a camera, not an error. Lumping it under `error` would have invited partners to wire alarm pipelines to benign cold-cache events.

Refined the contract before PR #60 lands on main:

| Path | Wire shape |
|---|---|
| Comparison ran, no motion | `{"stationary": true}` |
| Comparison ran, motion seen | `{"stationary": false}` |
| Benign skip — first call | `{"stationary": false, "reason": "no_previous_frame"}` |
| Benign skip — >20 min since last | `{"stationary": false, "reason": "previous_frame_stale"}` |
| Caller mistake | `{"stationary": false, "error": "no_camera_identifier"}` or `invalid_camera_id` |
| Backend failure | `{"stationary": false, "error": "s3_error"}` |

`stationary` is now always populated (default false on every non-success path) so consumers branch on it without null-handling. `reason` vs `error` is a benign-vs-actionable split — telemetry on the `error` rate is now a real health signal worth alerting on, instead of being dominated by cold-cache noise. Detect_motion was brand-new in PR #60 with no external consumers yet — right moment to harden the contract.

Also shipped: a side-by-side `/v5/test/motion` HTML demo page (local-dev only) for screenshot-ready cold→warm flow demos. PR #77 merged develop; live-verified on dev-api with the parking-lot JPEG (call 1: `reason: no_previous_frame`; call 2 32s later: `stationary: true`). ### Unplanned: PR #78 — per-detection `stationary_detection` + `stationary_object` tags via S3

User question: "is the setup we have capable of responding about stationary on a per-label basis? Are we tagging each object in the response?" — surfaced that the existing multi-frame `stationary_filter: tag` mechanism produced `stationary_detection` AND `stationary_object` per-detection tags, but the new single-frame `detect_motion` path produced zero per-detection tags. Built:
- Sibling `.detections.json` cache object next to the existing `.jpg` — same 20-min TTL, same lifecycle rule, existing IAM `*` wildcard covers both keys (no Terraform change needed)
- `_run_detect_motion` refactored to return `(result, motion_geometry, prior_detections)` 3-tuple

- Motion-demo page extended with a per-detection table + tag-count pills for screenshots

### §30 — cv2 `dst=` preallocation (sibling session, PRs open, soak pending)

Parallel session continued §30 from 2026-05-14's hotspot findings. Implemented the highest-payoff easy-fix candidate ([[2026-05-14_first-hotspot-findings#what-is-and-isnt-easy-to-fix]]): per-detector preallocated `cv2.foo(dst=…)` output buffers across `actuate-movement`'s frame-diff path, replacing fresh numpy allocations on every frame.

|---|---|---|---|
| Cumulative bytes allocated | 3,006 MB | 1,750 MB | **−42%** total |
| `actuate_movement` footprint | ~1,389 MB | ~190 MB | **−86%** |

**Correctness verification.** 14 unittest cases in `actuate-movement/tests/test_fdmd_buffer_equivalence.py` assert `np.array_equal` between the buffer-pool and fresh-alloc paths across `resize`, `preprocess_frame`, `get_threshold_img` (both `use_contours` modes × 4 sensitivities), buffer reallocation on shape change, buffer reuse on steady shape, the `update_background` regression test, and end-to-end `get_delta_noise` intermediates.

## Notes / Learnings

- **Contract bugs found via live wire trace, not via tests.** PR #77 was triggered when the verification output happened to show `{"error": "no_previous_frame"}` on the screen — that wording was clearly wrong but no test would have caught it. Lesson: rendering the actual response shape to a human (verification script output, screenshot in a PR comment) catches semantic issues that pass through every assertion. The motion-demo HTML page exists specifically to make this kind of inspection easy.

- **Same-day contract refinement is cheap when the feature is brand-new.** PR #77 changed the response field name (`error` → `reason` for two of five codes) — for a published API that would be a major churn event. Because `detect_motion` was unmerged-to-prod (parked in PR #60 → main), no external consumer was on the old shape yet. Window like this closes the moment the prod cutover lands.

- **Cache locality lets one design support two trigger paths.** The existing multi-frame `stationary_filter: tag` mode applies `stationary_detection` + `stationary_object` tags by comparing detections WITHIN a single request's frame sequence. PR #78 extends the SAME tags to single-frame `detect_motion` calls by sourcing the "prior frame" from S3 cache. Same tag values mean partners' existing tag-handling code works unchanged across both trigger paths.

- **Signal-config errors masquerade as silent breaks.** AP summary-rate signal was reading RED for 4 days, classified mentally as "another known-false RED signal" and ignored. It was actually a config typo (`container_name` exact-match vs `LIKE`-prefix). A signal-creation discipline rule (require a one-time non-zero validation against a known-healthy state) would have caught it at birth. Adding to §9.

- **Don't push WIP §29 branch.** Customer-model commit lives only locally on `feat/customer-deploy-branch-api` (commit `b0ce8cb0`). Has the migration + admin import + FK fix but no endpoint bodies. Resume handoff captures everything: [[2026-05-18_handoff-deploy-branch-phase1-resume]].

- **`makemigrations` proposes unrelated alters when other models drift.** Django bundled `timezone` choice alters on 7 models into the §29 migration output because a `django/zoneinfo` upgrade is pending elsewhere in the codebase. Hand-curated 0546 to exclude them — they belong in whoever's PR is doing the upgrade, not §29's.

- **CodeArtifact token name varies by repo.** `actuate-inference-api` pyproject names its index `codeartifact` → `UV_INDEX_CODEARTIFACT_*` env vars. `actuate_admin` names its index `private-registry` → `UV_INDEX_PRIVATE_REGISTRY_*`. Easy to confuse; documented in the §29 resume handoff.

- **The buffer-pool retain-across-frames gotcha (§30).** Almost shipped a silent correctness bug: `update_background` was doing `_, self.background_frame, _ = self.preprocess_frame(frame)`, and with the new buffer pool that means `background_frame` aliases `_buf_gray` — every subsequent `preprocess_frame` call overwrites the background, then `absdiff(gray, background_frame) = absdiff(gray, gray) = 0`, and motion detection silently dies. The user explicitly asked to verify 1:1 equivalence in all cases before claiming victory; auditing all callers of the modified methods is what surfaced this. Lesson: when introducing a buffer pool, audit every site where a return value gets stored in `self.*` — those are the retain-across-frames sites that need explicit `.copy()`.

- **The `cv2.foo(src) → cv2.foo(src, dst=…)` pattern is broadly underused.** Every cv2 op in the motion-detection path supported a `dst=` output buffer; ~50 LOC of preallocation replaced ~570 MB of churning numpy allocations per 60s. Look for this pattern any time a hot loop calls multiple cv2 ops on shapes that don't change. The same pattern likely applies elsewhere in `actuate-movement`'s second detection method (lines 650-710) and in `actuate_image_cache.PooledTTLImageCache`'s decode path — not in scope for #349 but worth a sweep after that lands.

- **Pre-commit + CodeArtifact = dual-index dance.** Pre-commit hook subprocesses create their own pip venvs and inherit `PIP_INDEX_URL` from env, but CodeArtifact's actuate repo doesn't proxy public PyPI (returns 404 on `setuptools`). Working combo: `PIP_INDEX_URL=https://pypi.org/simple/` primary, `PIP_EXTRA_INDEX_URL=https://aws:$TOKEN@.../pypi/actuate/simple/` for actuate packages. Plus `UV_INDEX_CODEARTIFACT_USERNAME=aws` + `UV_INDEX_CODEARTIFACT_PASSWORD=$TOKEN` for any uv-driven hook steps. Worth a snippet in onboarding docs.

- **`git worktree add` for multi-branch isolation when the primary working tree is mid-stream.** Both §30 PRs landed via worktrees at `/tmp/actuate-libraries-cv2` and `/tmp/vms-connector-cv2` so the original `feature/actuate-instrumentation-v1` branch's untracked harness work stayed untouched. Cleaner than `git stash` for selective-path workflows because nothing on the source branch changes — and worktrees auto-tear-down on `git worktree remove` once the branches merge or are abandoned.

- **Same commit, two changes, only one looked live.** First Eyeforce validation on the VCH branch showed the alert copy fix working but multi-frame scoring stayed at 1 frame/camera. Easy to misread as "deployer pulled an older SHA" because the symptoms were similar to the InvalidImageName tag bug we'd just fixed. Real cause: the drain loop's `while not empty: get()` raced against the puller still pushing — both changes WERE deployed but the multi-frame one had a timing bug. Lesson: when one half of a multi-change commit appears live and the other doesn't, the deploy is rarely the explanation — it's almost always a logic bug.

- **`while not empty: get()` is a race when there's a concurrent producer.** Standard `queue.Queue` pattern; biting because the puller thread runs the full ~2s clip and pushes frames over that whole window. The fix is structural — drain only after the producer thread has been joined — not a clever wait/timeout. Both `(a)` moving the drain post-join and `(b)` using blocking `get(timeout=X)` were on the table; once `(a)` is in, `(b)` becomes irrelevant because there's no producer left to race against.

- **`thread.join()` is the load-bearing synchronization in this codebase.** The puller thread's "I'm done" signal is its termination; anything that needs to see a complete queue from that puller must run after `thread.join()` returns. Worth keeping in mind for any future change that touches the healthcheck consumer loop.

- **Slashes in branch names break Docker tags silently.** `connector_deployer` wrote `feat/vch-multi-frame-quality-sampling` straight into the K8s image field. K8s rejects with `InvalidImageName`, `containerID: null`, no container ever runs — but NR shows the literal invalid tag in `container_image` so a casual NR-only check looks healthy. Two-step audit going forward: after any deployer claim of "deployed", check the *literal tag* in the cronjob spec and confirm no `/`. Issue filed against the deployer ([#171](https://github.com/aegissystems/connector_deployer/issues/171)).

- **No manual `kubectl create job --from=cronjob/...` on customer-bound patrols.** Tempting to short-circuit a 4h wait for the next natural cronjob tick, but those jobs fire detections into the customer's monitoring queue and can be actioned as live events. Wait for the natural tick even when it costs a cycle of validation latency.

- **AutoPatrol tier is a patrol-level property on EVERY tier-accepting API surface.** Initially I framed the tier change as just `raise_patrol_alert` — wrong. The Immix spec wording ("AI Provider should return … the highest Tier … regardless of whether any detections were actually raised") applies symmetrically to `get_patrol_stream` (where Immix uses tier for billing/routing the video clip). Both must carry the patrol's highest configured tier, not the firing detection's tier. Source of truth: `AutoPatrolConnectorConfig.configured_detection_codes` (new accessor) → `highest_tier_for(...)` (new helper).
