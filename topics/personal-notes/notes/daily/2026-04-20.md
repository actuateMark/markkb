## Notes / Learnings

- SessionStart hook output is injected into the starting session as context — Claude sees overlapping claims at cold-start without extra reads.
- Stop hook heartbeat uses `$PWD` from stdin payload as the claim key; sibling sessions keep each other's rows fresh automatically, no IPC needed. Works because real sessions are almost always in distinct repos (distinct CWDs).
- Build-side smoke test: the hook round-trip (claim → heartbeat update → startup prune) works end-to-end without Claude involvement. Race: if one session writes while another reads mid-splice, `fcntl.flock` handles it.
- **Subagents can fabricate convincing stack traces.** During the PR #1654 regression check, one nrql-investigator report claimed `KeyError: 'monitoring'` at a specific file:line in the autopatrol factory with a paraphrased Python traceback. Raw log query returned 0 hits for that exact fingerprint. The real bug was on `:rearch-dev` (stale branch with `actuate-config==1.4.4.dev6+ed.1`), not `:stage`. Two process failures combined: the agent paraphrased instead of quoting, and I narrowed too aggressively with `LIKE '%stage%'` before faceting image tags. Drove both the "require raw message evidence" and "facet-by-image first" guardrails in `/stage-regression-check` and `/bug-dive`.
- **Interpreting "new on tag" needs a cross-tag check.** When guardrail G3 flagged the streamId error as "0 occurrences pre-merge on :latest, 2 post-merge," my first read was "new regression." Real story: AP wasn't running on `:latest` pre-merge at all, so every AP error would look "new" by that test. Cross-tag FACET on the exact fingerprint (`:stage` had 65/7d, `:latest` had 2/72h) correctly classified it as pre-existing, newly exposed. The skill's rubric now needs the cross-tag step as a second stage to G3.
- **Field-name vs value distinction in .NET backend errors.** The `$.streamId` in Immix's 400 cites the JSON path in *our* payload, not their canonical `DeviceStreamID` property. Case-insensitive `System.Text.Json` binding means `streamId`, `deviceStreamId`, and `DeviceStreamID` all match the same server property. Useful to remember when parsing ASP.NET validation errors — the field name in the error isn't necessarily the server's internal name.
- **"Silent data-integrity fix" anti-pattern.** PR #1654 introduced `self.stream_id = uuid.uuid4().hex` in the new `PatrolCamera` class — fabricating a GUID client-side so the 400 stops happening. Problem: Immix accepts the call and records an alert against a streamId they've never seen. The 400 was the *correct* backend behavior; masking it creates an orphan-alert problem they can't trace back to any real stream. Removing this fabrication is on the connector-side cleanup list in GH#1656.
- **AutoPatrol + VCH local CHM tracking is preserved on Immix rejection.** In `vch_alert_sender.py:82-111`, `raise_patrol_alert` and `save_chm_issue` are in separate try/except blocks. A 400 on Immix dispatch doesn't prevent the `save_chm_issue` DAO write. So we always have our own record of the incident even when the outgoing alert fails. Useful framing for the Immix conversation: customer-visible data loss is limited to what the Immix-side UI shows; our internal audit trail is intact.
- **Signal overlap: `SiteDisabledOrDisarmed` failures are a cleanup-lambda signal in disguise.** Of the 10 recent `raise_patrol_alert failed` samples I grabbed for GH#1656, 5 had Immix response `Patrol id ... with status SiteDisabledOrDisarmed is not in the expected status of Started`. That's the same class of "site isn't coming online in Immix" signal our cleanup-Lambda initiative (§3 / `autopatrol_onboarder`) is built to detect — just through a different failure surface. Added as §3/§10 subtask: emit a distinct SQS event from the connector on `SiteDisabledOrDisarmed`, track in the cleanup Lambda's existing DDB counter with a longer window than "no patrols" (since disarmed ≠ deleted, can be transient business-hours state). Only soft-disable if site stays in that state continuously for weeks. Don't conflate with "no patrols" short-cycle logic.
- **"Camera offline" can mean "our TLS handshake fails."** Sentinel healthcheck values (resolution=0, fps=10000, is_blank_frame=true, broken_stream=true) present identically whether the camera is physically offline or we couldn't get past the cert-verify step. Easy to misread as "customer has broken cameras." Diagnostic key: check if WebSocket connect attempts exist in logs and whether they hit SSL errors — if yes, it's us, not them.
