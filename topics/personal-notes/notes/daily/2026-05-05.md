- **Lesson on signals.json deployment.** It's an Obsidian-Sync-shaped config (lives under `~/.claude/`, not the KB), but `scp` is the canonical sync. - **`error_rate` signal had a description-vs-NRQL mismatch.** Description claimed a `message LIKE '%ERROR%'` fallback; NRQL was `level='ERROR'` only. Per-the-NR-investigation that level isn't extracted from autopatrol-server text logs, the signal would have returned 0 forever even when real errors occurred. Fixed: `(level='ERROR' OR message LIKE '% - ERROR - %')` ‚Äî works today against the stdlib basicConfig prefix `<ts> - <logger> - ERROR - <body>`, stays correct once structured logging ships. **Self-rule:** when adding a signal whose description mentions a fallback path, write the NRQL to actually do the fallback, not just describe one.

- **Structured-logging value is smaller than yesterday's plan claimed.** Direct NR keyset check: 2,583 autopatrol-server log records / 24h, but `level` is queryable on only 3 (startup JSON lines from a library); `patrol_id`/`site_id`/`camera_id` are not top-level attrs at all. NR auto-detects JSON stdout, so emitting JSON would auto-promote `level` (real win), clean up `message` (real win), but **structured fields still need call-site changes** (`logger.info("event", extra={"patrol_id": pid})`) because they're inside f-strings. A formatter-only PR delivers the alerts-friendly `level` filter but not the "queryable patrol_id" claim.

- **Cross-stack tech debt.** vms-connector, actuate_admin, actuate-inference-api, actuate-libraries, autopatrol_onboarder all use stdlib `logging.basicConfig` text format ‚Äî none structured. One in-cluster precedent: `create-detection-window` emits JSON and has `level` clean across 100% of records. So adopting `python-json-logger` in autopatrol-server isn't first-mover territory; we'd be following an existing pattern. PR #2 is stack-wide debt, not a service-specific fix ‚Äî flagged in the upgrade plan note.
