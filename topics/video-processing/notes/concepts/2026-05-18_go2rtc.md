# go2rtc: multi-protocol camera streaming bridge

Camera-streaming application supporting dozens of formats and protocols. Distributed as a single Go binary, Docker container, or Home Assistant add-on. Best known as the streaming backbone behind the [[webrtc-deep-dive|WebRTC]] Camera HA integration.

## Surface

- **Default ports:** API server `:1984` (TCP), [[rtsp-deep-dive|RTSP]] `:8554` (TCP), [[webrtc-deep-dive|WebRTC]] `:8555` (TCP/UDP)
- **Container:** `alexxit/go2rtc` / `ghcr.io/alexxit/go2rtc` for 386, amd64, arm/v6, arm/v7, arm64
- **Config:** YAML `go2rtc.yaml` in cwd; minimal stream entry is `streams: { name: rtsp://... }`
- **HTTP API** intended for embedding into smart-home / control-plane platforms

## How it compares to MediaMTX

Smaller surface than [[2026-05-18_mediamtx]] — go2rtc's center of gravity is the Home-Assistant smart-camera bridge; MediaMTX targets the general media-server-and-proxy role with hot-reload, recording, Prometheus, etc. For an Actuate fleet-arch PoC the choice is roughly: HA-style simple bridge (go2rtc) vs. production-server feature surface (MediaMTX).

## Why it matters for Actuate

Reference architecture for "WebRTC-on-top-of-existing-RTSP-cameras." If a future puller variant needs to expose [[rtsp-deep-dive|RTSP]] cams as [[webrtc-deep-dive|WebRTC]] streams for a browser dashboard, go2rtc's wire format and config shape are a useful prior-art baseline even if we don't adopt the binary itself.

## Open questions

- Memory / connection-count limits per binary instance — is one-binary-per-site viable or do we need horizontal scale?
- HTTP API granularity — can we add/remove a single stream at runtime without rewriting the YAML?
