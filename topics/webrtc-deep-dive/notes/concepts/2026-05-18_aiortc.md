# aiortc: WebRTC and ORTC for Python

Pure-Python [[webrtc-deep-dive|WebRTC]] + ORTC implementation built on `asyncio`. API closely follows the JavaScript [[webrtc-deep-dive|WebRTC]] API but uses Pythonic constructs: promises → coroutines, events emitted via `pyee.EventEmitter`. Regularly interop-tested against Chrome and Firefox.

## Capabilities

- SDP generation and parsing
- ICE with half-trickle support + mDNS candidates
- DTLS key/cert generation, handshake, encryption / decryption (for SCTP)
- SRTP keying, encryption, decryption for RTP and RTCP
- Pure-Python SCTP implementation
- Data Channels (bidirectional)
- Audio: Opus, PCMU, PCMA (send + receive)
- Video: [[av1-vp9-future|VP8]], [[h264-deep-dive|H.264]] (send + receive)
- Audio / video / data-channel bundling
- RTCP reports including NACK and PLI for packet-loss recovery

## Why it matters for Actuate

Top candidate for any Python-side [[webrtc-deep-dive|WebRTC]] PoC — the connector is Python-heavy, and aiortc is the only mature pure-Python option. The [[h264-deep-dive|H.264]] send path matters specifically: most Actuate cameras already emit [[h264-deep-dive|H.264]], so re-encoding can be avoided if aiortc's RTP packetizer can pass our existing bitstreams through.

## Notable absences vs. native WebRTC stacks

- No [[h265-hevc-deep-dive|HEVC]] support (we have some [[h265-hevc-deep-dive|HEVC]] cameras — would need [[2026-05-18_pion-webrtc]] or a native bridge)
- No simulcast (single-quality streams only)
- Pure-Python SCTP/SRTP is slower than native at high stream counts — fleet-scale viability unverified
