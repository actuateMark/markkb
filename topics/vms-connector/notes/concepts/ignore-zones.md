# Ignore Zones

Ignore zones suppress detections in defined regions of a camera's field of view. They filter out nuisance sources -- busy roads, swaying trees, reflective surfaces -- that would otherwise generate false alerts. The system supports three zone types, each operating at a different layer of the [[pipeline-architecture]].

## Zone Types in settings.json

Camera settings are generated by [[admin-api/_summary|Actuate Admin API]] and stored in S3 as `settings.json`. The [[actuate-config]] library's `StreamDeploymentConfig` parses zone data from the feature block:

- **`polygonal_zones`** (feature-level): A list of polygon coordinate lists. Each polygon is an array of `[x, y]` points in normalised `[0, 1]` space (relative to frame dimensions). Applied post-inference to suppress detections whose bounding box centre falls within any polygon.
- **`motion_polygonal_zones`** (feature-level): Same format, but applied to the FDMD motion detection step via `IgnoreMotionPolygonalZonesStep`. Prevents motion regions in masked areas from even triggering inference.
- **Per-label `polygonal_zones`** (metric-level): Nested inside `raw_metrics` or specific metric blocks like `loiterer` or `vehicle_loiterer`. Allows different zone masks per detection product on the same camera -- a loitering zone can differ from an intruder zone.

The config parser sets boolean flags (`has_polygonal_ignore_zones`, `has_motion_ignore_zones`, `has_label_ignore_zones`, `has_loiterer_ignore_zones`) that pipeline steps check before activating zone filtering.

## PolyZoneCalc in actuate-filters

The [[actuate-filters]] library provides `PolyZoneCalc`, a static utility class built on Shapely 2.x vectorised geometry operations. It offers four operations:

- **`merge()`**: Unions multiple polygon lists into a single set of non-overlapping polygons. Used when combining zones from different sources (e.g., site-level and camera-level zones).
- **`intersect()`**: Computes the geometric intersection across polygon groups. Used when calculating the overlap between two zone sets.
- **`coverage_ratio()`**: Returns the fraction of the unit-square frame covered by the union of all zones, clipped to `[0, 1]`.
- **`is_fov_covered()`**: Returns `True` if zones cover more than a threshold (default 50%) of the frame -- a health check to detect misconfigured cameras where most of the view is masked.

Invalid polygons (fewer than 3 points, self-intersecting) are silently skipped with a log warning. All output coordinates are rounded to 4 decimal places.

The actual point-in-polygon test at runtime uses `matplotlib.path.Path.contains_point()` via the `ignore_polygonal_zones()` function and the `PolygonZoneFilterMixin`. The mixin caches converted paths per frame dimension to avoid recomputing on every detection.

## Flex Ignore Zones

The legacy model provides a single static set of ignore zones per camera. [[flex-ignore-zones]] (the dominant [[autopatrol/_summary|AutoPatrol (H1.2)]] workstream as of April 2026) extends this to **multiple named presets per camera**, with different presets activating per patrol schedule.

This solves the day/night problem: a loading dock ignored during business hours (constant legitimate movement) can be fully monitored at night by switching to a different preset. The settings generation pipeline resolves which preset applies when a schedule triggers, then generates the correct `polygonal_zones` array for the [[detection-pipeline|detection pipeline]].

The flex IZ system requires coordinated changes across the API layer (AUTO-500), frontend preset management (AUTO-427), schedule association (AUTO-446), and bulk update tooling (AUTO-424/493). The bulk update component is at "Ready to Deploy" status. The concept is directly portable to [[watchman/_summary|Actuate Watchman]], where schedule-based preset switching could eventually be automated by the Site Context Agent based on learned site patterns.
