# Actuate Admin API — endpoint catalog

High-level catalog of REST endpoints exposed by the [actuate_admin](https://github.com/aegissystems/actuate_admin) service. This note exists so other sessions can find the right endpoint + serializer fast without grepping the whole repo. **It is a snapshot, not the source of truth** — always verify against the repo before relying on an exact shape.

## Access

| Dimension | Value |
|---|---|
| Production base URL | `https://admin.actuateui.net/api/` |
| EU base URL | `https://admin.actuateui.eu/api/` |
| Auth — programmatic | `Authorization: Token <api-token>` (header). Token lives in AWS Secrets Manager at `prod/actuate/postgres` → key `api-token-prod`. Fetch with `scripts/fetch_admin_token.sh` in the `autopatrol_onboarder` repo or: `aws secretsmanager get-secret-value --secret-id prod/actuate/postgres --query SecretString`. |
| Auth — browser / Swagger UI | Session cookie. The Swagger UI (`/api/swagger/`, `/api/redoc/`) requires login and **does not accept Token auth** — opening it in a browser while logged-in is the way. |
| Pagination | DRF standard — `count / next / previous / results`, `?page=N`. Default page size 100. |
| Filtering | Per-viewset via django-filters. Most viewsets expose a `FilterSet` (grep the `_view.py` file for `filterset_class` or `filter_fields`). |
| Format | Mostly camelCase on the wire (the project uses explicit field aliases, not DRF's camelcase middleware globally). Snake_case is accepted by some legacy endpoints but **not all** — the AutoPatrolSchedule serializer is strict camelCase and silently drops snake_case keys. See [[2026-04-23_cleanup-rollout-day]] for the failure mode this caused during cleanup Lambda rollout. |

## Source of truth

- URL config: `actuate_admin/api/urls.py` (~290 lines; look for `router.register(...)` calls).
- Serializers: `actuate_admin/api/serializers/**/*.py` — directory-organized by feature.
- Views: same tree as serializers, file named `*_view.py` (viewset class) and `*_serializer.py` (serializer class).
- Swagger is generated by `drf-yasg` at `api/urls.py:126` via `get_schema_view(...)`. Same source reflects every endpoint the API exposes.

## High-level endpoint families (as of 2026-04-24)

Roughly 80 registered viewsets. Grouped by concern:

### Identity & access
- `auth/` — login + password reset
- `customer/` — customer (site) records
- `group/` — tenant group hierarchy
- `immix_credential/` — Immix API credentials per tenant
- `webhook_customer/` / `webhook/` / `webhook_type/` — outbound event webhooks
- `permission/` + `UserViewSet` — user + role management

### Devices & deployment
- `camera/` — camera records (deviceId, scheduleCameras, settings)
- `camera_type/`, `nvr_camera/`, `recording_server/` — device + NVR metadata
- `management_server/` — [[vms-connector|VMS connector]] deployment targets
- `storage/`, `configuration/`, `option/` — env + config flags
- `integration_type/` — enum-like list of supported VMS integrations (autopatrol, vch, axis, wisenet, mobotix, …)
- `schedule/` — generic healthcheck cron schedules (NOT AutoPatrol schedules)
- `schedule_location/` — geo-tagged schedule entries

### Patrol + cleanup (**autopatrol-specific, relevant for stale-schedule work**)
- `auto_patrol_schedule/` — **the primary schedule entity** for AutoPatrol + VCH. Detail of this one below.
- `auto_patrol_schedule/by_customer/<id>/` — list filtered to one customer
- `auto_patrol_schedule/by_tenant/` (POST) — sync-style bulk upsert; runs the `handle_schedule_status` chain that flips `is_deleted` on scheduleStatus transitions — see [[2026-04-23_cleanup-rollout-day]] §Path-A discussion

### Analytics & AI
- `ai_model/`, `sensitivity/`, `metric/`, `metric_label/`, `dwell/` — model + metric tuning
- `analytics/`, `clip/`, `images/` — analytics queries + media access
- `healthcheck/` — schedule-health check records (distinct from AutoPatrol health)

### Integration syncs (per-vendor)
- `aisync/`, `sentinelsync/`, `frontelsync/`, `umbosync/`, `yoursixsync/`, `axis/`, `mobotix/` — partner-specific sync entrypoints
- `alarm/` — alarm hub state

## AutoPatrolSchedule endpoint — the one that matters for cleanup work

### Path
`/api/auto_patrol_schedule/<admin_pk>/` (detail) — GET, PATCH.
`/api/auto_patrol_schedule/` (list) — GET, POST.

Explicit allow-list on the viewset: `http_method_names = ["post", "get", "patch"]`. **No DELETE** — that's deliberate, the soft-delete mechanism is `is_deleted=True` via Python-side transitions, not HTTP DELETE.

### Filters (query params on list endpoint)
- `scheduleId=<immix-uuid>` — exact match on the Immix scheduleId
- `customer=<int pk>` — customer foreign key
- `tenant=<uuid>` — tenant UUID (Immix-side)
- `disabled_by=<str>` — exact match; **the cleanup audit filter** (`?disabled_by=cleanup_lambda` returns all schedules the cleanup Lambda has touched)

### Serializer field catalog (read / write)
See `actuate_admin/api/serializers/integrations/autopatrol/autopatrol_schedule_serializer.py`. Fields are camelCase; common ones relevant for cleanup:

| Field | Type | R / W | Notes |
|---|---|---|---|
| `id` | int | R | admin primary key |
| `scheduleId` | UUID str | R/W | Immix schedule UUID |
| `scheduleStatus` | str | R/W | One of `Awaiting / Active / Suspended / Paused / Removed / Deleted`. Setting to `Deleted` / `Removed` / `Paused` / `Suspended` triggers the view's post-save hook → `model.deploy()` → `should_undeploy=True` → `model.undeploy()` (container teardown). |
| `disabledBy` | str | R/W | Provenance of auto-disable (e.g. `cleanup_lambda`, `user:alice@…`). |
| `disabledAt` | ISO8601 | R/W | Timestamp of auto-disable. |
| `reenabledBy` | str | R/W | Provenance of re-enable. |
| `reenabledAt` | ISO8601 | R/W | Re-enable timestamp. |
| `customer` | nested | R | `{id, name}` — site this schedule is under. |
| `devices` | nested list | R | Cameras assigned to this schedule. |
| `cronExpression`, `checksPerDay`, `cadence`, `cadenceUnit` | - | R/W | Schedule timing. |
| `createdDate`, `updatedDate` | ISO8601 | R | `updatedDate` auto-bumps on every save (including noop PATCHes). Silent-drop of unknown fields still bumps this — be careful. |

**NOT in the serializer (even though it's on the model):**
- `is_deleted` (boolean) — the soft-delete flag. You cannot toggle this via PATCH. The model flips it via `instance.delete()` / `soft_delete()` / `restore()` from Python code, OR via `by_tenant/` sync when incoming `schedule_status ∈ {Deleted, Removed}` triggers `handle_schedule_status` → `schedule.delete()`. For auto-cleanup we bypass by setting `scheduleStatus` to a `should_undeploy` value and relying on the view's existing deploy-chain to undeploy the container. Full rationale in [[2026-04-23_cleanup-rollout-day]] §Path-B.

### The PATCH trap (discovered 2026-04-23)
DRF's default is to silently drop unknown fields on PATCH. `updatedDate` still bumps via `auto_now=True` so the response looks successful. Implications:

- **Always verify disable PATCHes via a round-trip GET** — check that the field actually changed, not just that the PATCH returned 200. The cleanup Lambda (`cleanup_lambda.py:_disable_on_admin`) does this after PR #8.
- **camelCase is load-bearing** on this serializer — `disabled_by` (snake) silently drops; `disabledBy` (camel) sticks.
