# Python 3.15 `profiling.sampling` — Watchlist Brief

**Bottom line: not adoptable until ≥ October 2027.** New stdlib sampling profiler is genuinely new (not something we already use under another name), and architecturally close to py-spy / austin (the tool already wired into `cpu_profile.sh`). Worth tracking; not worth chasing.

## What it is

`profiling.sampling` (codename "Tachyon", PEP 799) is a statistical sampling profiler added to the Python stdlib in **3.15**. Contributed by Pablo Galindo and Laszlo Kiss Kollar. It periodically captures call-stack snapshots and aggregates them. Default 1 kHz, configurable up to a claimed 1 MHz. Modes: wall-clock (default), cpu-only, GIL-hold-time, exception-active.

Stdlib reorganization in 3.15 also puts `cProfile` under `profiling.tracing` — same code, new import path. Not relevant to adoption decisions.

## How it differs from what we already use

| Tool | We use it? | What `profiling.sampling` does differently |
|---|---|---|
| `cProfile` | available, rarely used (high overhead) | sampling vs deterministic — bounded overhead |
| `tracemalloc` | yes (`ACTUATE_TRACEMALLOC=1`) | unrelated — memory vs CPU |
| `py-spy` | yes (`cpu_profile.sh`) | same model; `profiling.sampling` is in-stdlib (no Rust binary), adds differential flamegraphs, async-aware mode, Gecko/Firefox JSON |
| `austin` | no | functionally equivalent; `profiling.sampling` supersedes |
| `memray` | yes (`memory_profile.sh`) | unrelated — memory vs CPU |
| `scalene` | no | scalene retains memory + GPU dimensions and line-level attribution; `profiling.sampling` is CPU-only |

**It is not a rename of anything we use.** It is a stdlib entrant in the same niche py-spy occupies.

## Notable features

- **Attach to live process** without restart: `python -m profiling.sampling attach <PID> -d 60`
- **Five output formats**: pstats text, collapsed stacks (flamegraph.pl-compatible), interactive HTML flamegraph, Gecko JSON (Firefox Profiler), source-annotated heatmap HTML, plus a compact binary capture.
- **Differential flamegraphs**: `--diff-flamegraph baseline.bin` — red/blue/gray/purple diff. Directly comparable across runs. **This is the one feature with no good open-source equivalent today.**
- **Async-aware**: reconstructs call stacks across `await`. Incompatible with `--native` / `--all-threads` / CPU / GIL modes.
- **"Zero measurable overhead on the target"** — sampler is a separate process reading memory externally.
- **`--blocking` mode** pauses target during each sample (consistency at the cost of jitter). Off by default. **Do not use in production.**

## Why we can't adopt now

1. **Python 3.15 final = October 2027.** ~17 months away.
2. **Connector is on Python 3.12** (verified via `test_vms/__pycache__/*.cpython-312.pyc`).
3. **Profiler/target must match minor version exactly.** A 3.15 sampler cannot attach to a 3.12 process — the offsets it reads are version-specific.
4. **Same ptrace requirement as py-spy.** No change to K8s deployment friction.

The third point is the hard blocker. Even if we wanted to use 3.15-on-3.15 ahead of the GA, we'd need to migrate the connector through 3.13 → 3.14 → 3.15 first.

## Migration considerations (for the eventual upgrade)

`docs/OPTIMIZED-CONNECTOR.md` already includes evaluation sections for 3.13 and 3.14 (per-interpreter GIL, no-GIL builds, free-threaded mode). 3.15 watchlist items to fold into that work when it happens:

- **`profiling.sampling` replaces py-spy** in the connector toolchain, no third-party binary needed.
- **Differential flamegraphs in CI** become viable — `--binary` capture before + after a PR, diff in PR comment.
- **Async-aware mode** would be the cleanest way to profile `AsyncInferencePool`'s coroutine churn — current py-spy view aggregates all the asyncio frames into noise.
- **Gecko JSON output** opens the door to uploading captures to a hosted Firefox Profiler instance for share-with-link reviews.

## What to do now

- **No code changes.** Keep using py-spy + memray + tracemalloc + jemalloc breakdown.
- **Track the connector's Python version migration** as the gate. When 3.15 is on the runtime roadmap, revisit this note and rewire `cpu_profile.sh`.
- **One thing to validate before 3.15 lands**: confirm that whatever K8s capability gymnastics we land for the py-spy sidecar (roadmap item 7) translates 1:1 to `profiling.sampling`. They use the same ptrace mechanism, so this should be free, but worth a 30-min check when the sidecar is designed.
