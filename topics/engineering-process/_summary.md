## Key Principles

> **Top principle — regression prevention is the #1 operational priority for every launch.** Above all else, every launch's priority is to avoid regressions and to catch them as soon as possible when they do occur. The [[2026-04-23_dashboard-sketch|operational dashboard]] is the forcing function: every release runs it, every morning reviews it, and no launch is considered verified until the dashboard is GREEN against the deployed component for at least one cron cycle / one real-traffic window. See [[2026-04-23_postmortem-onboarder-healthcheck]] for the incident this came from — 47 hours of silent failure is unacceptable.

The remaining principles support that top priority:

1. **Design for easy monitoring** — every feature, every proposal, every sketch answers "how do I know this is working in prod?" before sign-off. `Errors=0` is not a health signal; activity-marker log lines + downstream side effects are. Per-repo signal set lives in each repo's `CLAUDE.md`; cross-repo aggregation in the dashboard ([[mark-todos]] §9).
2. **Release is verified, not declared** — every merge that hits prod/stage runs the repo's acceptance-criteria check against the live system, then the dashboard end-to-end. CI-green is necessary but never sufficient. See [[2026-04-23_release-acceptance-criteria]].
3. **Plan before code** — architecture decisions and design review happen before implementation. Monitoring hooks are part of the plan, not an afterthought.
4. **KB-driven context** — always check the KB before starting; always update it after finishing.
5. **Security at every boundary** — validate all inputs, check RBAC before validation, never leak internal state.
6. **Test what you build** — unit tests + local integration test + full regression before merge.
7. **Document what you ship** — API docs, backend docs, KB synthesis notes, all updated before deploy.
8. **Review intensely** — lint, security audit, code review, then review again.
